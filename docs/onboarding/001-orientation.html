온보딩 001 - 오리엔테이션과 지도

eks-bootstrap 를 제로에서 빌드하면서 (1) Terraform 을 제대로 이해하고 (2) SOC 통제를 코드에 박는 여정의 출발점

대상: 000 - Terraform 기초로 HCL 블록, state/backend, plan 읽는 법, root/module 분리를 이미 뗀 나 자신.
목표: 이 문서를 읽으면 (1) 이 레포가 왜 modules/ + environments/ + infra/ 로 나뉘는지, (2) SOC 통제가 어느 시점, 어느 코드 줄에 박히는지 (Tier 1/2/3), (3) 한 번의 변경이 plan 문서 -> apply -> 검증 -> 정합성으로 어떻게 흐르는지를 말로 설명할 수 있다.
읽는 법: 이 문서는 시리즈의 지도다. 코드가 없어도 지금 읽을 수 있다. 002 부터는 빌드 단계마다 하나씩 써 나간다 (6장 매핑 표).
표기 규칙: 이 레포는 ASCII + 한글만 쓴다 (이모지 / 유니코드 화살표 금지). 화살표는 ->. 근거는 documentation-style.md.

왜 이 시리즈가 따로 필요한가. 이전 스터디 문서는 이미 다 만들어진 레포의 코드를 읽는 법을 가르쳤다. 그건 1단계였다. 이 레포(eks-bootstrap)는 정반대 상황이다 - .tf 가 아직 한 줄도 없는 그린필드다. 여기서 필요한 건 "남의 코드를 읽는 능력"이 아니라 "구조를 설계해서 직접 짜는 능력" + "그 코드에 SOC 통제를 언제 박을지 판단하는 능력"이다. 이 시리즈는 그 두 능력을 빌드 단계와 1:1로 붙여 키운다.

0. 이 문서 시리즈는 무엇인가

온보딩 시리즈(docs/onboarding/NNN-*.html)는 이 레포를 빌드하는 학습 동반 문서다. 운영용 문서(docs/conventions/, docs/security/ 등)를 대체하지 않고, 그 문서들을 읽고 코드를 짤 수 있도록 왜 / 언제 / 어떤 순서로 를 먼저 깐다.

문서성격언제 보나
이 시리즈 (docs/onboarding/)학습 / 빌드 동반지금. 그리고 각 빌드 단계 직전
CLAUDE.md정체성 / invariants이 레포의 규칙이 헷갈릴 때
docs/conventions/규칙 (naming/tag/변수/버전/문서)코드를 작성 / 리뷰할 때
docs/security/보안 baseline + SOC 2 / ISO / ISMS-P어떤 통제를 박을지 판단할 때
docs/plans/apply 전 변경 계획서자원을 만들거나 바꾸기 직전
docs/next-steps.md세션 간 인계 (live 진척)"지금 어디까지 했나"가 궁금할 때

진척 상태(완료된 step 체크)는 이 문서가 아니라 next-steps.md 가 들고 있다. 이 문서는 지도라 잘 안 바뀐다.

1. 두 목표

이 레포에서 얻으려는 것은 크게 둘이다. 시리즈의 모든 문서는 이 둘을 동시에 겨눈다.

목표 1 - Terraform 을 제대로목표 2 - SOC 적용 bootstrap
핵심 질문"이 코드가 왜 이렇게 생겼나"를 설명할 수 있나"이 통제가 왜 지금, 여기 박히나"를 설명할 수 있나
다루는 것문법(HCL) + 아키텍처(모듈 경계, 의존, state) + 설계 결정SOC 2 통제 -> Terraform 코드 줄로 번역, Tier(immutable) 판단
합격 신호plan 출력을 읽고 destroy/replace 를 알아본다Tier 1 을 day 1 에 빼먹지 않는다

두 목표는 사실 한 몸이다. "SOC 를 코드에 박는다"는 건 결국 Terraform 으로 kms_key, encryption_config, enabled_cluster_log_types 같은 자원/인자를 정확한 순서로 쓰는 일이다. Terraform 을 모르면 SOC 를 코드로 못 박고, SOC 요구를 모르면 어떤 인자를 켜야 할지 모른다. 그래서 한 문서 안에서 같이 배운다.

2. 참고 레포 vs 이 레포 - "읽기"에서 "짓기"로

Terraform 학습은 보통 이미 다 만들어진 인프라를 읽으며 한다 (000 도 완성된 코드를 예로 든다). 이 레포는 정반대 - 그린필드다. 이 차이를 모르면 "코드 예시가 왜 없지?"에서 막힌다.

보통의 학습 / 예제이 레포 (eks-bootstrap)
코드 상태이미 다 만들어진 인프라를 읽음그린필드 - .tf 0 줄, docs/ 만 있음
주된 활동남이 만든 코드를 읽고 검증구조를 설계해서 직접 짓고, SOC 를 박음
주요 능력plan 읽기, 아키텍처 해석모듈 경계 설계, 생성 순서 설계, 통제 시점 판단
첫 use-case(다양)Temporal on EKS (Phase 1 PoC)

그래서 이 001 에는 실제 .tf 예시가 거의 없다. 아직 코드가 없으니까. 코드 예시는 003 (backend) 부터 빌드와 함께 등장한다. 001 의 임무는 "어떤 땅에, 어떤 순서로, 어떤 규칙으로 지을지"의 지도를 그리는 것이다.

3. 이 레포의 정체성 - 3분할 구조

이 레포는 세 종류의 Terraform 코드 영역으로 나뉜다. 디렉토리 이름만 외우지 말고 각자 역할이 왜 다른지를 잡아야 한다.

영역무엇인가핵심 제약

왜 3분할인가

이 분할은 첫 코드 한 줄 쓰기 전에 ADR 로 결정됐다. 두 개의 결정이 핵심이다.

  • ADR 0001 - modules / environments 분리: 이 레포는 두 역할을 동시에 한다 - (1) 외부 fork 대상 generic 모듈, (2) 내가 실제로 띄워보는 sandbox. 환경을 1급 시민으로 두고(environments/dev/), 모듈은 거기서 호출하는 building block 으로 둔다. 환경 추가 = 디렉토리 복제. fork 소비자는 modules/ 만 가져간다.
  • ADR 0003 - backend bootstrap 은 local state: state 를 S3 에 두기로 했지만, 그 S3 버킷은 누가 만드나? Terraform 으로 만들고 싶은데 그 Terraform 의 state 는 또 어디? 이 순환을 infra/tf-backend/ 라는 작은 stack 이 local state 로 1회 apply 해서 끊는다. 이후 environments/<env>/ 는 partial backend config 로 그 S3 를 backend 로 쓴다.

분할의 진짜 효과는 "강제"다. modules/ 와 environments/ 를 물리적으로 나누면, "모듈에 환경값/시크릿을 박지 말라"는 규칙이 구조적으로 강제된다 (모듈 안에는 tfvars 가 없으니 박을 데가 없다). 이건 terraform-variables.md 의 invariant 와 직결된다 - 모듈은 입력만 받는 "테스트 가능한 부품"이어야 한다.

4. SOC 가 코드에 박히는 법

목표 2 의 핵심. 이 레포는 SOC 2 Type II (Security TSC) 를 1차 프레임워크로 잡고, 그 기술 통제 대부분을 Terraform 으로 자동 충족하는 것을 노린다. ISO 27001:2022(글로벌 사업)와 ISMS-P(한국 시장)는 보조 매핑이다.

여기서 가장 중요한 통찰: 모든 통제가 같은 시점에 박히는 게 아니다. 어떤 건 "day 1 에 안 박으면 나중에 켤 수 없거나 자원을 재생성해야" 하고(immutable), 어떤 건 운영 중 언제든 추가할 수 있다. 이 레포는 그걸 Tier 1 / 2 / 3 로 분류한다.

Tier 1 / 2 / 3 - 언제 박나

Tier의미시점예
Tier 1immutable - 생성 시에만 가능하거나, 나중에 바꾸면 재생성 비용이 매우 큼첫 apply (day 1)S3 Object Lock, EKS Secret KMS envelope, Aurora storage_encrypted, VPC CIDR, IMDSv2
Tier 2운영 중 추가 가능하나, evidence 일관성 위해 빨리 박는 게 좋음Phase 1 후반 ~ Phase 2CloudTrail multi-region + LFV, log retention 365일, CMK rotation, mTLS, ESO
Tier 3운영 시작 후 자유롭게 추가Phase 3+GuardDuty / Security Hub, WAF, Pod Security, NACL, Multi-AZ

Tier 1 을 day 1 에 빼먹으면 비싸다. 예: S3 버킷의 Object Lock 은 버킷 생성 시에만 켤 수 있다 (나중에 못 켬). EKS Secret 봉투 암호화(KMS envelope)는 클러스터 생성 시에만 설정된다 - 나중에 켜려면 클러스터를 재생성해야 한다. Aurora storage_encrypted 는 생성 후 토글 불가다. 그래서 security-baseline.md 는 Tier 1 immutable 항목 11개를 추려, 첫 plan 부터 강제 점검하게 한다.

이게 목표 1(Terraform)과 어떻게 만나는가: Tier 1 항목 하나하나가 결국 특정 자원의 특정 인자다. 예를 들어 -

5. Phase 로드맵

빌드는 한 번에 끝나지 않는다. 세 Phase 로 나뉘고, 각 Phase 가 audit 후보 시점과 맞물린다.

Tier 와 Phase 는 다른 축이다. Tier = 그 통제를 나중에 추가하면 얼마나 비싼가 (immutable 성질, 4장). Phase = 언제 실제로 하나 (시간/성숙도). Tier 1 이 Phase 1 에 강제되는 건 immutable 이라 day 1 아니면 재생성/불가여서지, Tier 가 Phase 의 다른 이름이라서가 아니다. 그래서 한 Tier 가 여러 Phase 에 걸친다 - 특히 Tier 2 는 Phase 1 후반부터 Phase 2 까지 갈라진다 (가벼운 건 이미 Phase 1 baseline 에, 무거운 건 Phase 2). Tier 3 은 Phase 3+. 단 감사로그 버킷 Object Lock 처럼 분류는 Tier 3 라도 버킷 생성(Phase 2) 시점에 박아야 하는 예외가 있다 (생성 후 못 켬).

  • Phase 1 (PoC) Tier 1 전부 + 가벼운 Tier 2 - 지금 여기. VPC + EKS + RDS Aurora Serverless v2 + Temporal (PG visibility, mTLS/인증은 아직 X). Tier 1 immutable baseline 이 박히고, 가벼운 Tier 2(commit-gates, EKS API 제한, RDS force_ssl, RDS public access=false, IRSA, 필수 태그, retention 90일)도 함께 들어간다.
  • Phase 2 (Staging) 무거운 Tier 2 - state 환경별 분리, mTLS, ESO, ArgoCD(패키지만), 관측성, log retention 365일 상향, CMK rotation, VPC Flow/ALB logs, pgAudit. CloudTrail multi-region + LFV 는 Phase 1 후반에 당겨도 됨. Type I audit 후보 시점.
  • Phase 3 (Prod) Tier 3 - Multi-AZ provisioned, DR, NetworkPolicy, Pod Security, GuardDuty/Security Hub, WAF, NACL. Phase 2 + 6~12개월 운영 후 Type II audit 후보.

근거: CLAUDE.md Phase 로드맵 + soc2-compliance.md audit timeline.

6. 빌드 순서 - 온보딩 문서 매핑

Phase 1 은 정해진 순서가 있다 (모듈은 아무 순서로나 못 만든다 - state backend 가 먼저 있어야 환경이 apply 되고, VPC 가 있어야 EKS 가 뜬다). 각 step = 별도 plan 문서(docs/plans/YYYYMMDD-<slug>.md) + 이 시리즈의 온보딩 문서 하나.

plan 은 안전, apply 는 사람. terraform plan / validate / fmt -check 는 아무것도 안 바꾸므로 마음껏 돌린다. apply 만 실제 자원을 바꾸며, 이건 사람이 직접 실행한다. 그리고 plan 이 못 잡는 위험(apply/runtime 함정)이 있다 - 그래서 S4 의 검증 명령을 apply 후 실제로 돌려 S7 에 기록하는 단계가 따로 있다. 함정 목록은 versions.md "Apply/Runtime 함정 노트".

정합성 체크가 이 레포의 핵심 invariant 다. 같은 사실(통제 번호의 의미, 버전, 수치, 항목명)이 여러 파일에 등장하면, 한 곳을 고칠 때 grep 으로 전부 대조해 함께 고친다. 외부 표준/버전 같은 사실 주장은 추정하지 말고 출처를 확인한다. 수정이 2개 이상 파일에 걸치거나 사실 주장을 건드리면 /ralph self-review 루프를 돈다. 기계적 체크(fmt/validate/tflint/tfsec/시크릿)는 pre-commit 훅이 담당하고, 정합성 체크는 훅이 못 잡는 의미적 일관성을 담당한다. 근거: CLAUDE.md 운영 invariants.
