1. Terraform 이 뭔가, 왜 쓰나

Terraform 은 인프라를 코드로 선언하는 도구다 (IaC, Infrastructure as Code). "EKS 클러스터 1개, 노드 2대, DB 1개를 원한다"를 텍스트 파일에 적으면, Terraform 이 클라우드 API 를 호출해서 그 상태를 맞춰준다.

콘솔 클릭과 뭐가 다른가

콘솔에서 클릭Terraform 코드
기록누가 뭘 바꿨는지 모름git 에 diff 로 남음 (PR 리뷰 가능)
재현stage/prod 만들 때 또 클릭코드 복사 + 값만 변경
실수되돌리기 어려움apply 전에 plan 으로 미리 확인
리뷰불가능변경을 코드로 검토 후 승인

핵심 개념: 선언형 (declarative). "이렇게 해라" (명령형) 가 아니라 "최종 상태가 이래야 한다" (선언형) 를 적는다. Terraform 이 현재 상태와 비교해서 차이만 실행한다. 같은 코드를 두 번 apply 해도 이미 맞으면 아무 일도 안 일어난다 (idempotent, 멱등).

2. 5분 정신 모델: Terraform 이 일하는 방식

딱 세 가지만 기억하면 된다.

  1. 코드 (desired state) -- .tf 파일. 내가 원하는 최종 모습.
  2. state (현재 알고 있는 상태) -- terraform.tfstate. Terraform 이 "지금 실제로 뭐가 있는지" 기억하는 장부.
  3. 실제 클라우드 (real world) -- 진짜 자원.

terraform plan 은 이 셋을 비교해서 "코드대로 만들려면 무엇을 더하고 / 바꾸고 / 지워야 하는가"를 보여준다. terraform apply 는 그 계획을 실제로 실행하고 state 를 갱신한다.

flowchart LR
  CODE["코드 .tf
(원하는 상태)"]
  STATE["state
(현재 장부)"]
  AWS["실제 클라우드
(진짜 자원)"]
  CODE -- "terraform plan
(비교만, 변경 없음)" --> DIFF["차이 (plan)"]
  STATE --> DIFF
  DIFF -- "terraform apply
(실제 실행)" --> AWS
  AWS -- "state 갱신" --> STATE

plan 은 안전, apply 는 위험. plan 은 아무것도 바꾸지 않는다 (읽기 전용, 마음껏 돌려도 됨). apply 만 실제 자원을 바꾼다. 안전한 운영에서는 apply 를 사람이 직접 실행하고, 그 전에 반드시 plan 으로 영향을 확인한다.

3. HCL 기본 블록

Terraform 코드는 HCL (HashiCorp Configuration Language) 로 쓴다. 문법은 단순하다. 거의 모든 것이 block 형태다.

block_type "label1" "label2" {
  argument = value
  nested_block {
    key = value
  }
}

실제로 만날 블록은 사실상 7종뿐이다. 이것만 알면 대부분의 코드가 읽힌다.

resource 와 data

resource = "이 자원을 만들어라". 가장 많이 보는 블록이다.

resource "aws_s3_bucket" "state" {
  bucket = var.bucket_name        # 인자 (argument)

  lifecycle {
    prevent_destroy = true        # 실수로 지워지지 않게 보호
  }
}
  • aws_s3_bucket = 자원 타입 (provider 가 제공). 첫 단어 aws 가 provider.
  • state = 내가 붙인 이름 (코드 안에서만 쓰는 별명). 다른 곳에서 aws_s3_bucket.state.id 로 참조.
  • bucket, lifecycle = 그 자원의 설정.

data = "이미 존재하는 것을 조회만 해라" (만들지 않음).

data "aws_caller_identity" "current" {}   # 현재 계정 ID 등을 조회

data "aws_vpc" "existing" {
  id = var.vpc_id                          # 기존 VPC 를 조회만 (Terraform 이 만든 게 아님)
}

resource vs data 한 줄 요약. resource 는 Terraform 이 소유하고 관리한다 (만들고 바꾸고 지움). data 는 남이 만든 것을 읽기만 한다. Terraform 밖에서 만들어진 자원(계정, 리전, 기존 네트워크)은 data 로 참조하고 절대 건드리지 않는다.

variable 와 output

variable = 코드의 입력값. 환경마다 달라지는 값을 코드에 박지 않고 변수로 뺀다.

variable "log_retention_days" {
  description = "CloudWatch Logs 보존 기간 (일)."   # 설명 (필수)
  type        = number                               # 타입
  default     = 90                                   # 기본값 (있으면 선택 입력)
}

variable "cluster_name" {
  description = "EKS 클러스터 이름. env 별 (tfvars 주입)"
  type        = string
  # default 없음 -> 값을 안 주면 에러. 반드시 tfvars 에서 줘야 함 (의도적 안전장치)
}

변수 값은 terraform.tfvars 에서 채운다:

cluster_name = "eksbs-dev"

코드에서 변수를 쓸 때는 var.이름:

retention_in_days = var.log_retention_days
name              = "${var.cluster_name}-sg"   # "${...}" = 문자열 안에 값 끼워넣기

output = 코드의 출력값. 한 모듈이 만든 결과를 다른 모듈/사람에게 알려준다.

output "cluster_endpoint" {
  description = "EKS API endpoint"
  value       = aws_eks_cluster.this.endpoint
}

module 와 locals

module = 재사용 가능한 코드 묶음을 불러 쓰기. 잘 짠 아키텍처의 핵심이다. 환경(root) 코드는 거의 전부 module 호출이다.

module "eks" {
  source = "../../modules/eks-cluster"   # 어떤 모듈을 쓸지 (경로)

  cluster_name    = "eksbs-dev"           # 모듈에 넘기는 입력 (그 모듈의 variable)
  cluster_version = "1.35"
}

모듈의 출력은 module.이름.출력명 으로 받는다. 이렇게 모듈끼리 연결된다:

module "node_group" {
  source       = "../../modules/eks-node-group"
  cluster_name = module.eks.cluster_name      # eks 모듈의 output 을 node-group 모듈 input 으로
}

locals = 코드 안에서만 쓰는 임시 계산값 (변수와 비슷하나 외부 입력 아님).

locals {
  name_prefix = "eksbs-${var.environment}"
}
# 사용: local.name_prefix

terraform / provider 블록 = 어떤 버전, 어떤 클라우드를 쓸지 선언 (4장, 9장에서 자세히).

블록한 줄 의미보통 두는 파일
resource자원을 만든다거의 모든 .tf
data기존 것을 조회만 한다data.tf (보통 root)
variable입력값variables.tf
output출력값outputs.tf
module코드 묶음 불러쓰기main.tf (root)
locals내부 계산값main.tf / locals.tf
terraform/provider버전/클라우드 설정provider.tf, versions.tf

4. state 와 backend (가장 헷갈리는 부분)

terraform.tfstate 는 Terraform 이 "내가 관리하는 자원이 지금 뭐가 있는지"를 적어둔 JSON 장부다. 이 파일이 곧 진실의 원천이다. plan 이 비교하는 "현재 상태"가 바로 이거다.

state 는 신성하다. state 가 깨지거나 사라지면 Terraform 은 "이 자원들을 내가 만든 적 없다"고 착각해서 또 만들려 하거나, 반대로 못 지운다. 그래서 (1) 함부로 손으로 편집하지 않고, (2) 팀이 공유하며, (3) 잠금(lock)을 건다.

remote backend

state 를 각자 노트북에 두면 팀이 충돌한다. 그래서 원격 저장(remote backend)한다. AWS 에서는 S3 가 표준이다.

terraform {
  backend "s3" {
    bucket       = "eksbs-tfstate"            # state 를 저장하는 S3 버킷
    key          = "environments/dev/terraform.tfstate"
    region       = "ap-northeast-2"
    use_lockfile = true                       # S3 native lock (동시 apply 방지)
    encrypt      = true
  }
}
  • S3 버킷: 팀 전원이 같은 state 를 본다.
  • use_lockfile = true: 누군가 apply 중이면 잠금 파일을 만들어 다른 사람의 동시 apply 를 막는다 (Terraform 1.10+ 의 S3 native lock. 예전엔 DynamoDB 테이블을 따로 썼지만 이제 불필요).
  • 암호화 + 버전 관리: 버킷이 암호화 + 버전 보관이면 이전 state 로 복구 가능.

닭과 달걀 문제 (bootstrap)

"state 를 담을 S3 버킷"은 누가 만드나? Terraform 으로 만들고 싶지만, 그 Terraform 의 state 는 또 어디에 두나? 이 순환을 풀려고, 그 버킷만 만드는 별도의 작은 root 를 두고 그것만은 로컬 state 로 1회 apply 한다. 버킷에는 prevent_destroy = true 를 걸어 실수로도 못 지우게 한다. 이 레포에서는 ADR 0003 이 이 결정을 다룬다 (003 에서 직접 만든다).

5. plan 읽는 법 제일 중요

딱 하나만 제대로 배운다면 이것이다. terraform plan 출력을 읽을 줄 알면 사고의 대부분을 막는다. plan 은 각 자원 앞에 기호를 붙인다.

기호뜻위험도
+create (새로 만든다)보통
~update in-place (그 자리에서 수정)보통
-destroy (지운다)주의
-/+destroy and then create replacement (지우고 다시 만든다 = 교체)위험
<=read (data source 조회. 변경 아님)없음

가장 무서운 줄: -/+ (replace). 이건 자원을 지웠다가 새로 만든다. EKS 클러스터, RDS, EBS 볼륨 같은 데서 이게 뜨면 데이터 유실 / 다운타임이다. plan 출력에 forces replacement 또는 must be replaced 라는 문구가 같이 나온다. 이 문구를 보면 무조건 멈추고 왜 교체가 필요한지 따져라. 도구가 "그냥 apply 하면 됩니다"라고 해도, replace 가 있으면 사람이 직접 판단해야 한다.

실제로 어떻게 보이나

  # module.eks.aws_eks_cluster.this will be updated in-place
  ~ resource "aws_eks_cluster" "this" {
        id      = "eksbs-dev"
      ~ version = "1.34" -> "1.35"        # ~ = 그 자리에서 버전만 올림 (안전)
    }

  # aws_security_group.admin must be replaced
-/+ resource "aws_security_group" "admin" {
      ~ name = "old-name" -> "new-name"   # forces replacement (위험! SG 가 지워졌다 생김)
    }

Plan: 1 to add, 1 to change, 1 to destroy.

맨 끝 줄 Plan: X to add, Y to change, Z to destroy 가 요약이다. destroy 숫자가 0 이 아니면 항상 한 줄 한 줄 확인한다.

"No changes." 변경이 없으면 No changes. Your infrastructure matches the configuration. 가 나온다. 코드를 정리(리팩터)만 했는데 이 메시지가 뜨면 = 동작은 그대로라는 뜻 (좋은 신호). 값을 변수로 빼는 작업 같은 게 그렇다.

6. 작업 사이클 + 자주 쓰는 명령

명령은 두 부류로 나뉜다.

읽기 전용 (안전)상태 변경 (사람만)
plan, validate, fmt -check, show, state list, outputapply, destroy, import, state rm/mv, taint
# 읽기 전용 -- 마음껏 돌려도 안전
terraform -chdir=environments/dev init       # 최초 1회 (provider/backend 준비)
terraform -chdir=environments/dev validate   # 문법 검사
terraform -chdir=environments/dev plan        # 변경 미리보기 (제일 자주)
terraform -chdir=environments/dev fmt -check  # 포맷 검사

# 상태 변경 -- 사람이 자기 환경에서 직접만
terraform -chdir=environments/dev apply

-chdir=... 는 "그 폴더에서 실행"이라는 뜻. apply 하는 단위(root)가 그 폴더다.

안전한 변경 흐름. 코드를 바꾸면 -> fmt / validate 로 형식·문법 확인 -> plan 으로 영향 확인 (특히 destroy/replace) -> 리뷰 -> 사람이 apply -> 결과를 실제 명령으로 검증. 이 레포는 이 흐름을 plan 문서 + 정합성 체크로 강제한다 (001 의 작업 사이클 참고).

7. root 와 module

Terraform 폴더는 두 종류로 나뉜다: root (apply 하는 단위) 와 module (재사용 부품).

environments/          # root 모듈들 (apply 단위)
  dev/                 # 한 환경. 여기서 terraform apply 함
    provider.tf        # provider + backend + 버전 (terraform/provider 블록)
    data.tf            # 기존 자원 조회 (data 블록, root 전용)
    main.tf            # 모듈 호출 + 연결 (아키텍처의 중심)
    variables.tf       # 입력 변수 정의
    terraform.tfvars   # 환경별 실제 값 (하드코딩은 여기만)
    outputs.tf         # 출력

modules/               # 재사용 부품 (직접 apply 안 함. root 가 호출)
  vpc/
  eks-cluster/
  ...

module 폴더의 공통 규칙

각 모듈 폴더는 관심사별로 파일을 쪼갠다 (main.tf, variables.tf, outputs.tf, versions.tf ...). 그리고 다음을 모듈 안에 두지 않는다: provider, backend, terraform.tfvars, 그리고 외부 상태 조회용 data (계정/리전/기존 자원). 이것들은 root 가 주입한다 (이유는 8장).

8. 모듈 작성 / 재사용

모듈은 "입력(variables) -> 자원 생성 -> 출력(outputs)" 의 블랙박스다. 잘 만든 모듈은 dev/stage/prod 가 같은 코드를 쓰고 값만 다르게 준다.

# 모듈의 인터페이스 = variables(입력) + outputs(출력)
# 호출 측 (root):
module "vpc" {
  source     = "../../modules/vpc"
  cidr_block = var.vpc_cidr      # 입력 주입
  region     = var.aws_region
}
# 모듈 안 (modules/vpc/variables.tf):
variable "cidr_block" { type = string }
variable "region"     { type = string }
# 모듈은 var.region 을 쓰지, 직접 data 로 리전을 조회하지 않는다.

왜 모듈 안에서 외부 조회를 금지하나. 모듈이 안에서 "현재 리전/계정"을 직접 조회하면, 그 모듈은 호출 환경에 숨은 의존성이 생겨 재사용/테스트가 어려워진다. 그래서 계정 ID, 리전, 기존 자원 같은 외부 컨텍스트는 root 가 조회해서 변수로 넣어준다. 모듈은 "입력만 받는 테스트 가능한 부품"이어야 한다. 이 레포의 규칙은 terraform-variables.md.

9. 버저닝

버전이 안 맞으면 어제 되던 게 오늘 깨진다. Terraform 은 여러 층의 버전을 핀(pin) 한다.

terraform {
  required_version = ">= 1.11"      # Terraform 자체 버전 (S3 native lock 은 1.10+ 필요)

  required_providers {
    aws        = { source = "hashicorp/aws",        version = "~> 6.0" }
    tls        = { source = "hashicorp/tls",        version = "~> 4.0" }
    kubernetes = { source = "hashicorp/kubernetes", version = "~> 3.0" }
    helm       = { source = "hashicorp/helm",        version = "~> 3.0" }
  }
}
표기뜻예
>= 1.111.11 이상 아무거나Terraform 본체
~> 6.06.x 까지 허용, 7.0 은 막음 (pessimistic)AWS provider
정확한 값딱 그 버전만chart_version 등
  • .terraform.lock.hcl: provider 의 정확한 버전 + 체크섬을 잠가, 팀 전원이 동일 provider 를 쓰게 한다. terraform init 이 만든다. git 에 커밋한다.
  • EKS 쿠버네티스 버전, Helm chart 버전 등도 핀 대상이다. 이 레포의 실제 핀 값과 변경 절차는 versions.md 가 single source of truth.

버전 올릴 때. provider major 업(예: aws 6 -> 7), 쿠버네티스 마이너 업은 동작이 바뀔 수 있으므로 항상 plan 으로 영향 확인 + 별도 계획서로 진행한다. 특히 plan 이 통과해도 apply/runtime 에서 깨지는 "silent breakage" 가 있어, 메이저 변경은 공식 release note 를 직접 확인한다.
