1. 왜 규칙이 코드보다 먼저인가

혼자 쓰는 코드라도 규칙을 먼저 정하는 이유는 셋이다.

  • 리뷰 왕복을 줄인다. 이름/태그/변수 형식이 정해져 있으면 리뷰는 "형식이 맞나"가 아니라 "이 변경이 안전한가"에 집중한다.
  • fork 가능성을 지킨다. modules/ 는 외부 fork 대상(ADR 0001)이라, 조직 고유값이 코드에 박히면 못 쓴다. 규칙이 이걸 구조적으로 막는다.
  • 일관성이 곧 감사 증거다. 모든 자원이 같은 태그/이름 규칙을 따르면, "관리되지 않는 자원이 없다"를 기계적으로 보일 수 있다 (SOC 2).

2. 네이밍 resource-naming.md

AWS 자원 이름 포맷은 하나다:

eksbs-<env>-<component>[-<suffix>]
조각의미값
eksbseks-bootstrap 식별 prefix고정 (fork 시 자기 prefix 로 교체)
<env>환경dev / staging / prod
<component>컴포넌트 (단어 1개)vpc, eks, rds, temporal ...
<suffix>같은 component 가 여러 개일 때만-a, -writer ...

예시: eksbs-dev-vpc, eksbs-dev-eks (cluster name), eksbs-dev-rds-temporal (Temporal 용 Aurora cluster).

Terraform identifier 는 다른 규칙

위는 AWS 자원 이름이고, 코드 안의 Terraform 라벨은 별개다:

  • 리소스 라벨: 모듈 안에 단일 인스턴스면 this, 그 외엔 의미 있는 이름. (예: resource "aws_eks_cluster" "this")
  • 모듈 호출 이름은 역할 기준: module "vpc", module "temporal_irsa".
  • 모두 snake_case (Terraform 관례).

금지. 대문자 / 공백 / 한글 / 특수문자 (DNS-1035 위반 가능), 64자 초과 (RDS 등 자원 제한), 환경명 없는 이름(다중 환경에서 충돌). 이름에 환경이 없으면 staging/prod 를 만들 때 부딪친다.

3. 태깅 tagging.md

모든 AWS 자원에 필수 태그 4종을 붙인다. 자원마다 손으로 붙이지 않고, provider 의 default_tags 로 한 번에 박는다.

키값 예시의미
Projecteks-bootstrap레포 식별 (fork 시 변경)
Environmentdev환경
ManagedByterraform수동 자원과 구분 (drift 식별)
OwnerGitHub 핸들 / 팀명책임자

선택 태그: Component(eks/rds/temporal), CostCenter(learning/platform).

# environments/dev/provider.tf
provider "aws" {
  region = var.aws_region

  default_tags {
    tags = {
      Project     = "eks-bootstrap"
      Environment = var.environment   # "dev"
      ManagedBy   = "terraform"
      Owner       = var.owner
    }
  }
}

자원별 추가 태그는 모듈 안에서 합친다: merge(var.tags, { Component = "eks" }).

태그에 시크릿/개인정보 금지. 태그는 IAM 정책 밖에서도 광범위하게 노출된다 (비용 리포트, 콘솔, API). 비밀번호/토큰/주민번호 같은 건 절대 태그에 넣지 않는다.

4. 변수 + 파일 구조 terraform-variables.md

000 에서 본 root/module 구분을 이 레포의 실제 파일 배치로 구체화한다.

모듈 (modules/<name>/)

modules/<name>/
|-- main.tf          # 주요 리소스
|-- variables.tf     # 모듈 입력
|-- outputs.tf       # 모듈 출력
|-- versions.tf      # required_version, required_providers
`-- README.md        # 용도 + input/output 표 + 예시 호출

자원이 많아지면 iam.tf, security_group.tf, addons.tf 등으로 분리. 한 파일 200~300줄 넘기 전에 분리 고려.

환경 (environments/<env>/)

environments/<env>/
|-- backend.tf       # S3 backend partial config
|-- provider.tf      # AWS/Kubernetes/Helm providers + default_tags
|-- data.tf          # 기존 자원 참조 (선택)
|-- variables.tf     # 환경 변수 선언
|-- terraform.tfvars # 환경별 값 (git 추적, 시크릿 X)
|-- main.tf          # modules/* 호출
`-- outputs.tf       # 환경 출력

변수 명명

  • snake_case.
  • bool 은 enable_* 또는 *_enabled.
  • 리스트는 복수형 (subnet_ids).
  • AWS 자원 ID 변수는 <resource>_id (예: vpc_id).

시크릿은 terraform.tfvars 에 박지 않는다. tfvars 는 git 에 추적된다 (환경 config 용). 시크릿은 Phase 1 에서는 SSM Parameter Store, Phase 2 부터 ESO(External Secrets Operator) 경유로 분리한다. commit-gates 의 gitleaks 가 실수를 한 번 더 막는다 (7장).

5. 모듈 / 환경 경계 제일 중요

이 레포에서 가장 자주 어기는 규칙이자 모듈 재사용성의 핵심이다. 무엇이 어느 방향으로 흐르는지 고정한다.

flowchart LR
  ENV["environments/dev
(root)"]
  MOD["modules/*
(부품)"]
  ENV -- "AWS 식별자(vpc_id 등),
환경 이름, 태그,
환경별 설정값" --> MOD
  MOD -- "후속 모듈이 쓸
ID / endpoint / ARN" --> ENV
  • 환경 -> 모듈: AWS 식별자(vpc_id, cluster_name), 환경 이름, 태그, 환경별 설정값.
  • 모듈 -> 환경: 후속 모듈이 필요로 하는 ID / endpoint / ARN (output).
  • 모듈 내부 결정(예: subnet CIDR 계산)은 모듈 안에서 끝낸다. 환경은 결과만 받는다.

왜 이 경계가 SOC 와도 연결되나. 모듈에 환경값/시크릿을 박지 않는 규칙(ADR 0001 + 000 의 8장)은 "재사용 가능한 부품"을 위한 것이지만, 동시에 시크릿이 generic 코드에 새어들지 않게 한다. 경계가 명확하면 "어디에 무엇이 있는지"가 분명해지고, 그게 곧 감사 가능성이다.

6. AWS 계정 사전조건 (step 0) 코드 아님

Terraform 작업 전에 AWS 계정에 박혀 있어야 하는 1회성 설정이다. 코드가 아니라 console/CLI 로 한다. Tier 1 항목을 포함하므로 빼먹으면 나중에 비싸다.

항목적용 방법Tier왜
EBS default encryption ON (region)aws ec2 enable-ebs-encryption-by-defaultT1retroactive 불가. 안 박으면 그 이후 만든 모든 볼륨이 미암호화로 시작
S3 public access block (account-level)aws s3control put-public-access-blockbaseline실수로 만든 bucket 도 자동 차단
Root account MFAconsole 설정T1SOC 2 CC6.2 기본. day 1 이 가장 안전
CloudTrail multi-region trail + LFVTerraform 또는 consoleT2region default trail 로는 부족. evidence 일관성 위해 일찍 권장
시간 동기화AWS 기본 (Amazon Time Sync)-추가 작업 불필요. 단 audit 시점에 명시

EBS default encryption 은 순서가 중요하다. 이건 "이 시점 이후 생성되는 볼륨"에만 적용된다 (retroactive 불가). 그래서 어떤 노드/볼륨도 만들기 전에, 즉 EKS node group(step 5)은 물론 그 어떤 자원보다 먼저 켜야 한다. 그래서 step 0 (맨 처음)이다. 근거: security-baseline.md 사전조건.

7. commit-gates (step 0.5) commit-gates.md

모든 commit 이 통과해야 하는 pre-commit 훅이다. 목적은 (1) 시크릿이 git history 에 들어가는 것 차단, (2) 포맷/문법/보안 위반 사전 차단. tf-backend(step 1)보다 먼저 깐다 - 첫 커밋부터 게이트가 걸려야 하니까.

카테고리훅막는 것
시크릿 탐지gitleaksAWS key, GitHub PAT, private key, generic API token
시크릿 탐지detect-private-keyRSA/SSH/PGP private key 파일
Terraformterraform_fmt비포맷 코드
Terraformterraform_validateinvalid HCL
Terraformterraform_tflint비추천 패턴 / deprecated 인자
Terraformterraform_trivytfsec rule 위반 (public S3, 0.0.0.0/0 SG, 평문 시크릿 등)
일반 위생end-of-file-fixer, trailing-whitespace, check-merge-conflict, check-yaml줄바꿈 / 공백 / merge 마커 / 깨진 YAML

설정 위치

  • .pre-commit-config.yaml (루트) - 훅 정의 + 버전 핀. 핀의 single source of truth (versions.md "Commit-gate 훅").
  • .tflint.hcl (루트) - 지금은 번들 terraform ruleset 만 활성. AWS ruleset(tflint-ruleset-aws)은 첫 AWS 모듈 작성 시(step 3) 추가.

로컬 설치 (개발자 1회)

훅이 호출하는 도구(gitleaks, tflint, trivy, terraform)는 pre-commit 이 자동 설치하지 않는다 - system PATH 의존:

# macOS (Homebrew). terraform 은 이미 설치돼 있다고 가정.
brew install pre-commit gitleaks trivy
brew install terraform-linters/tap/tflint   # tflint 는 homebrew-core 에 없음 -> 공식 tap

pre-commit install            # git hook 등록 (1회)
pre-commit run --all-files    # 전체 1회 실행 (.tf 없으면 terraform_* 훅은 skip)

함정. brew install tflint 단독은 실패한다 (homebrew-core 에 없음). 반드시 terraform-linters/tap/tflint. 또한 brew install A B tflint 처럼 묶으면 잘못된 formula 하나가 batch 전체를 중단시키므로 tflint 는 줄을 분리한다.

Bypass 금지. --no-verify 는 쓰지 않는다. 예외는 훅 자체에 버그가 있을 때만이고, 그때도 docs/plans/ 에 사유 / 범위 / 복구 절차를 적은 plan 을 먼저 쓴다. SOC 2 매핑: CC8.1 (변경 관리) - 코드 품질 / 시크릿 게이트가 적용됐다는 증거. (ISMS-P 2.8 자동 충족.) CI 백업(GitHub Actions 에서 동일 훅 재실행)은 Phase 1 후반에 도입.
